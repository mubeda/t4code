T4 Connect Architecture Overview

T4 Connect links a local T4Code environment to a cloud account, provisions a managed Cloudflare tunnel, and lets web or desktop clients connect through relay-issued, proof-bound credentials. Native phone clients and platform alert delivery are no longer part of this repository.

Relay

Stores account-owned environment links, provisions managed endpoints, verifies Clerk tokens and DPoP proofs, and issues short-lived connection material.

Environment Server

Signs link, status, and connect proofs with the local environment key and mints the final environment access token after relay authorization.

Clients

Web and desktop clients authenticate with Clerk, mint relay DPoP tokens, then talk directly to the managed environment endpoint for normal HTTP and WebSocket traffic.

High-Level Flow

Local server -> Relay: link environment with signed proof
Relay -> Cloudflare: reserve tunnel and hostname
Web/Desktop client -> Relay: request status or connection with DPoP
Relay -> Managed endpoint: verify environment health or request credential
Managed endpoint -> Web/Desktop client: normal HTTPS and WSS traffic

Trust Boundaries

Boundary Transport Authorization Purpose
CLI or bundled server -> relay link management HTTPS Clerk bearer token plus local environment-signed link proof Create, refresh, or revoke a cloud environment link.
Web/Desktop client -> relay protected endpoints HTTPS Relay-issued DPoP token plus per-request DPoP proof List environments, check status, and request remote connections.
Relay -> managed environment endpoint HTTPS over the managed tunnel Short-lived relay-signed JWT request proof Request signed health data or ask the environment to mint a credential.
Remote client -> environment HTTPS and WSS over the managed tunnel Environment-issued access token bound to the client DPoP key Use the normal environment HTTP APIs and WebSocket sessions.

1. Link

The CLI or local server gets a relay challenge, signs it with the environment key, and sends the proof to POST /v1/client/environment-links. The relay stores the link and ensures a managed endpoint is available when managed tunnels are enabled.

2. Discover

A signed-in client calls GET /v1/client/environment-links to list linked environments and display their endpoint metadata.

3. Status

The client calls POST /v1/environments/:environmentId/status. The relay verifies client DPoP, asks the managed endpoint for a signed health response, and returns the environment status.

4. Connect

The client calls POST /v1/environments/:environmentId/connect. The relay asks the environment to mint a short-lived credential bound to the client DPoP public key, then the client connects directly to the environment.

Endpoint Summary

Endpoint Authentication Use
GET /v1/client/metadata Public Discover relay issuer, supported client IDs, and supported scopes.
POST /v1/client/dpop-token Clerk token exchange payload plus DPoP proof Issue a relay DPoP access token for supported client scopes.
POST /v1/client/environment-link-challenges Clerk bearer Create a challenge that a local environment signs before linking.
GET /v1/client/environment-links Clerk bearer List the signed-in user's linked environments.
POST /v1/client/environment-links Clerk bearer plus environment-signed proof Create or update an environment link.
DELETE /v1/client/environment-links/:environmentId Clerk bearer Revoke an environment link for the signed-in user.
POST /v1/environments/:environmentId/status Relay DPoP Request and validate a signed environment health response.
POST /v1/environments/:environmentId/connect Relay DPoP Request and validate an environment bootstrap credential.
